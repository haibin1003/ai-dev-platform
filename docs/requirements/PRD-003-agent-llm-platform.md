## 1. 背景与目标

### 1.1 背景

当前平台已完成工作流引擎核心（PRD-001）和前端可视化基础（PRD-002 Phase 2），但存在以下关键缺失：

- Agent 与 LLM 未解耦，无法灵活切换底层模型
- 前端工作流编辑器功能残缺（工具栏崩溃、节点端口异常、状态映射不一致）
- 缺乏项目级代码仓库上下文，Agent 在空目录中工作
- 单一 Agent 执行模式，无法支持多角色协作
- 缺少个人助理（持久型 Agent）能力
- 未考虑 SaaS 多租户隔离

### 1.2 目标

构建一个**AI 原生研发协同平台**，实现：

- 🤖 **智能体工厂**：支持创建、配置、管理多种类型的 AI Agent
- 🔌 **LLM 即插即用**：统一接入 Claude、Kimi、OpenAI、DeepSeek 等模型
- 📁 **工程上下文感知**：项目绑定 GitHub 仓库，Agent 基于真实代码库工作
- 🔄 **多 Agent 协作编排**：架构师 → 开发 → 测试 → 审核，流水线化执行
- 🦞 **个人助理（龙虾）**：持久型 Agent，主动推送、长期记忆
- 🏢 **SaaS 就绪**：多租户、平台 Agent 市场、用量计费

### 1.3 目标用户

| 用户角色 | 核心诉求 |
|---------|---------|
| 开发团队负责人 | 标准化研发流程、降低协调成本 |
| 技术项目经理 | 可追溯的需求交付、自动化报告 |
| AI Agent 开发者 | 可配置的 Agent 模板、可观测的执行过程 |
| 平台管理员（SaaS） | 租户隔离、资源配额、平台级 Agent 配置 |

## 2. 术语定义

| 术语 | 定义 |
|------|------|
| **LLM Provider** | 大语言模型提供商，如 Anthropic、Moonshot、OpenAI |
| **Agent 模板** | Agent 的蓝图定义，包含角色 Prompt、工具集、LLM 配置、约束条件 |
| **Agent 实例** | 基于模板创建的运行中进程（任务型）或持久服务（持久型） |
| **持久型 Agent** | 长期运行的 Agent（如个人助理"龙虾"），有记忆、可主动推送 |
| **任务型 Agent** | 为单次任务创建的 Agent，任务完成即消亡（进程结束 + 上下文释放） |
| **沙箱（Sandbox）** | 隔离的文件系统工作目录，Agent 只能在此范围内操作 |
| **工程上下文包** | 包含代码仓库、项目规范、需求文档、历史摘要的完整工作上下文 |
| **产物（Artifact）** | Agent 执行的输出，包括代码 diff、测试报告、架构文档、审核意见等 |
| **编排（Orchestration）** | 平台按 DAG 顺序调度多个 Agent 协作完成复杂任务 |

### F-1: LLM 接入管理

| 属性 | 内容 |
|------|------|
| 描述 | 统一管理大语言模型提供商的连接配置 |
| 优先级 | P0 |
| 类型 | 纯后端 |

**验收标准**：

6. 支持 Provider 级别的开关（启用/禁用）

### F-2: Agent 模板管理

| 属性 | 内容 |
|------|------|
| 描述 | 定义和管理 Agent 的模板，作为创建 Agent 实例的蓝图 |
| 优先级 | P0 |
| 类型 | 前后端 |

**验收标准**：
1. 平台预置标准模板：
   - `claude-code-developer` — Java/Spring 后端开发
   - `kimi-code-developer` — Python/Node 开发
   - `opencode-developer` — 通用代码开发
   - `architect` — 系统架构设计
   - `test-engineer` — 测试用例设计与执行
   - `code-reviewer` — 代码审查
   - `security-auditor` — 安全审计
   - `doc-writer` — 技术文档编写
2. 用户可基于平台模板克隆创建自定义模板
2. **Agent 人格 Prompt 分层**（借鉴 IDENTITY/SOUL/AGENTS/USER/TOOLS 分层）：
   - `IDENTITY.md` — "我是谁？" 名字、身份、风格、表情、头像
   - `SOUL.md` — "你是什么样的人" 核心信念、做事方式、风格偏好
   - `AGENTS.md` — "每次会话做什么" 会话流程、记忆机制、工具使用规范
   - `USER.md` — "关于你的主人" 用户名字、称呼、时区、偏好
   - `TOOLS.md` — "本地笔记" 速查表、常用命令
3. 模板包含字段：
   - 基本信息：ID、名称、描述、图标
   - 运行时：Agent CLI 类型（claude/kimi/opencode）、启动参数
   - LLM 配置：关联的 Provider、默认模型、Fallback 模型
   - **人格配置**：IDENTITY / SOUL / AGENTS / USER / TOOLS 五层 Prompt（每层可独立编辑）
   - 工具集：允许使用的工具列表（file_read/file_write/bash/git/mvn/npm 等）
   - 约束条件：超时时间、最大 Token 数、是否允许网络访问、允许操作的目录
   - **Claude Code 详细配置**：PermissionMode、AllowedTools、Timeout、Env、ExtraArgs、Cwd
4. 模板版本管理：支持版本化，可回滚到历史版本
5. 模板状态：DRAFT / PUBLISHED / DEPRECATED

**前端要求**：
- Agent 模板列表页（卡片式展示，带搜索和分类筛选）
- 模板详情页（展示 Prompt、工具集、约束条件）
- 模板编辑页（支持 Prompt 实时预览）

### F-3: Agent 实例生命周期

| 属性 | 内容 |
|------|------|
| 描述 | 管理 Agent 实例的创建、执行、终止、休眠、消亡全生命周期 |
| 优先级 | P0 |
| 类型 | 前后端 |

**验收标准**：

#### 3.1 任务型 Agent 生命周期
```
创建 ──► 克隆模板（Replica）──► 启动进程 ──► 挂载沙箱 ──► 注入上下文 ──► 执行 ──► 提取产物 ──► 归档 ──► 消亡
```
1. **创建实例时**：基于模板 **深拷贝**（Replica）生成独立配置，自动分配唯一 ID 和 Code
2. **自动配置隔离**：Replica Agent 自动设置工作目录（Cwd/WorkDir）、ForkSession = true（独立会话）
3. **血缘追踪**：记录 `shadowFrom`（来源模板 ID），便于清理和溯源
4. 启动进程：使用 ProcessBuilder / Docker 启动真实 Agent CLI 进程
5. 挂载沙箱：为实例分配独立工作目录 `/sandboxes/{tenant}/{project}/{task-id}/`
6. 注入上下文：将项目代码（git clone）、规范文档、需求 PRD 放入沙箱
7. 执行：通过 SSE/WebSocket 实时透传 Agent 的 stdin/stdout/stderr
8. 提取产物：任务结束后自动提取 git diff、报告文件、日志归档
9. 归档：对话历史、执行记录存入数据库
10. 消亡：终止进程、释放资源、清理沙箱（保留 7 天用于审计）

#### 3.1a 启动僵尸清理（Stale Cleanup）
- 平台启动时自动扫描所有 RUNNING/PENDING 状态的 AgentInstance
- 检查对应进程是否存在：不存在 → 标记为 FAILED，清理沙箱
- 检查最后更新时间：超过 30 分钟未更新 → 标记为 FAILED，清理分身 Agent
- 保证服务器异常关闭（kill -9、OOM）后的状态一致性

#### 3.1b 项目级并发限制
- 每个 Project 可配置 `maxConcurrentAgents`（默认 3）
- 创建 AgentInstance 前检查该项目当前 RUNNING/PENDING 数量
- 超过限制时返回 `ErrMaxConcurrentAgentsReached`，拒绝创建
- 防止单个项目耗尽平台资源

#### 3.2 持久型 Agent 生命周期（龙虾）
```
创建 ──► 启动常驻进程 ──► 长期运行 ──► 空闲休眠 ──► 唤醒服务 ──► 归档销毁
```
1. 创建后长期保持进程或会话
2. 空闲时进入休眠（释放 CPU，对话历史存入 Redis）
3. 新任务或用户消息时唤醒（恢复对话历史）
4. 支持主动推送任务进度到用户前端
5. 用户可手动销毁或平台定期回收长期空闲实例

#### 3.3 通用管理功能
1. 实例列表：展示所有运行中/已结束的实例
2. 实时监控：CPU/内存占用、当前执行步骤、剩余 Token
3. 强制终止：用户可随时 kill 实例进程
4. 日志回放：查看已结束实例的完整执行日志

### F-4: 项目与代码仓库管理

| 属性 | 内容 |
|------|------|
| 描述 | 项目维度管理，绑定 GitHub 仓库，维护工程上下文 |
| 优先级 | P0 |
| 类型 | 前后端 |

**验收标准**：
1. 项目 CRUD：名称、描述、关联的 GitHub 仓库地址
2. GitHub OAuth 授权：用户授权平台访问其仓库（只读或读写）
3. 仓库同步：首次绑定时自动 clone 到平台存储；后续支持手动/自动同步
4. 工程上下文包自动生成：
   - `repo/` — 代码仓库（支持稀疏检出，大仓库可选子目录）
   - `docs/conventions.md` — 项目编码规范（从仓库根目录的 AGENTS.md/.cursorrules 读取）
   - `docs/history/` — 相关历史需求的摘要
5. 分支策略：Agent 工作时在 feature 分支提交，验收后由平台创建 PR 合入主分支
6. 项目级 Agent 配置：可为不同项目指定默认使用的 Agent 模板

### F-5: 需求管理与验收

| 属性 | 内容 |
|------|------|
| 描述 | 需求录入、拆分、指派 Agent、验收标准管理 |
| 优先级 | P0 |
| 类型 | 前后端 |

**验收标准**：
1. 需求录入：支持文本/Markdown 格式录入需求描述
2. 验收标准：为需求定义可量化的验收条件（如"单测覆盖率 ≥ 80%"）
3. 需求拆分：持久型 Agent（龙虾）辅助拆分需求为子任务
4. Agent 指派：为每个子任务指定执行 Agent（架构师/开发/测试/审核）
5. 产物收集：每个 Agent 执行完成后必须提交标准化产物
6. 人类验收：平台汇总所有产物（代码 diff + 报告），人类确认后合入
7. 需求状态：PENDING → ANALYZING → DESIGNING → DEVELOPING → TESTING → REVIEWING → ACCEPTED / REJECTED

### F-6: 多 Agent 协作编排

| 属性 | 内容 |
|------|------|
| 描述 | 基于 DAG 的工作流引擎升级，支持多 Agent 分阶段协作 |
| 优先级 | P0 |
| 类型 | 前后端 |

**验收标准**：
1. 工作流节点类型升级：
   - `AGENT_EXECUTION`：Agent 执行单元（替代原有的 `TASK`）
   - `HUMAN_APPROVAL`：人工审批节点（需要人类确认才能继续）
   - `CONDITION`：条件分支（根据上游产物决定分支）
   - `PARALLEL`：并行执行多个 Agent
2. 产物传递：上游 Agent 的输出自动作为下游 Agent 的输入
   - 产物类型标准化：`CODE_DIFF`、`ARCH_DOC`、`TEST_REPORT`、`AUDIT_REPORT`
3. 节点配置：
   - 选择 Agent 模板
   - 输入映射（引用上游产物或需求参数）
   - 输出产物定义
   - 超时设置
4. 执行策略：
   - 串行：按拓扑排序依次执行
   - 并行：无依赖的节点同时执行
   - 条件：根据表达式判断分支
5. 人工干预：Agent 执行过程中可以向人类提问（通过 WebSocket 推送），人类回答后继续

   - 按 Agent 执行次数计费
   - 按 LLM Token 消耗计费

### F-9: 个人助理（龙虾）

| 属性 | 内容 |
|------|------|
| 描述 | 持久型个人助理 Agent，支持长期记忆和主动推送 |
| 优先级 | P1 |
| 类型 | 前后端 |

**验收标准**：
1. 创建助理：用户可创建自己的持久型 Agent，命名（如"龙虾"）、设定性格、专长

   - 用户偏好（技术栈、编码风格、沟通方式）

   - 报告汇总（"汇总今天的所有执行结果"）
5. 生命周期：用户可手动销毁；平台对长期闲置（30天无交互）的助理自动休眠

### F-11: 场景模板（Scenario Template）

| 属性 | 内容 |
|------|------|
| 描述 | 预置的开发协作流水线模板，用户一键应用到项目，无需从零搭建 DAG |
| 优先级 | P0 |
| 类型 | 前后端 |

**验收标准**：
1. **内置场景模板**（参考 Heartbeat Scenario 设计）：
   - `feature-dev` — 需求分析 → 架构设计 → 编码实现 → 单元测试 → 代码评审
   - `bug-fix` — 问题分析 → 定位根因 → 修复代码 → 回归测试 → 验证关闭
   - `code-review` — 静态分析 → 安全扫描 → 质量评审 → 报告生成
   - `doc-sync` — 检测变更 → 补充文档 → 更新 API 文档 → 发布
2. 每个场景项包含：
   - 名称、执行间隔（分钟）、需求类型、使用的 Agent 模板
   - Prompt 模板（支持 `${project.git_repo_url}` / `${requirement.title}` 等变量）
3. **平台预置 + 用户自定义**：平台提供标准场景，用户可克隆修改
4. **一键应用**：选择场景 → 应用到项目 → 自动生成 Heartbeat 定时任务
5. **场景与 DAG 的关系**：简单流程用场景模板，复杂流程用 DAG 编辑器

**前端要求**：
- 场景市场页面（卡片展示，一键应用）
- 场景预览（展示步骤列表和每个步骤的 Prompt）
- 应用确认弹窗（展示将创建哪些定时任务）

### F-12: Prompt 模板引擎

| 属性 | 内容 |
|------|------|
| 描述 | 结构化 Prompt 构建系统，支持变量替换和分层组装 |
| 优先级 | P0 |
| 类型 | 纯后端 |

**验收标准**：
1. **变量替换**：支持 `${project.name}` / `${project.git_repo_url}` / `${requirement.title}` / `${workspace.path}` / `${timestamp}` / `${state.current}` / `${state.ai_guide}`
2. **分层组装**：按 PLATFORM → IDENTITY → SOUL → AGENTS → USER → TOOLS → TASK 顺序组装最终 Prompt
3. **执行契约注入**：在任务型 Agent 的 Prompt 中自动注入：
   - "当前阶段：xxx，下一步动作：执行状态转换进入下一阶段"
   - "必须执行命令，不输出解释性长文本"
   - "每个 PHASE 完成后必须执行 ON_PHASE_COMPLETE 命令"
   - "命令失败自动修复重试（最多3次）"
4. **调度员模式**：心跳/巡检类 Agent 的 Prompt 自动注入：
   - "你是调度员，负责编排任务而非直接修改代码"
   - "严禁直接修改任何源代码文件"
   - "所有代码改动必须通过创建新需求交由 CodingAgent 完成"
5. **模板缓存**：编译后的模板缓存，避免每次重复解析

---

### F-13: 冷却机制（Cooldown）

| 属性 | 内容 |
|------|------|
| 描述 | 防止 Agent 对同一对象（Issue/PR/需求）反复操作 |
| 优先级 | P1 |
| 类型 | 前后端 |

**验收标准**：
1. **平台层冷却**：Requirement / Artifact 记录 `lastAgentInteractionAt`（最后 Agent 交互时间）
2. **检查规则**：Agent 执行前检查目标对象最近 3 小时内是否已有 Agent 操作，有则跳过
3. **Prompt 层冷却**：在 Prompt 中明确要求 Agent 自检时间戳（兜底）
4. **可配置**：冷却时间可配置（默认 3h，可调整为 1h/6h/24h）
5. **冷却跳过记录**：跳过操作记录到日志，前端可查看"本次跳过原因"

---

### F-14: Webhook 自动触发

| 属性 | 内容 |
|------|------|
| 描述 | GitHub Webhook 事件自动触发 Agent 任务 |
| 优先级 | P1 |
| 类型 | 前后端 |

**验收标准**：
1. GitHub Webhook 配置：项目绑定 webhook URL，支持 push / pull_request / issue 事件
2. 事件映射：不同事件触发不同场景（push → doc-sync / PR open → code-review）
3. Webhook 签名验证：验证 GitHub 签名，防止伪造请求
4. 事件日志：记录所有收到的事件和触发的任务
5. 失败重试：事件处理失败时自动重试 3 次

## 5. 非功能需求

### 5.1 性能

| 指标 | 目标 |
|------|------|
| Agent 进程启动时间 | < 10s（含沙箱创建、代码 clone） |
| 首次输出延迟 | < 5s（Agent 启动后到首条 stdout） |
| SSE 推送延迟 | < 100ms（后端收到输出到前端展示） |
| 工作流保存 | < 500ms |
| 页面加载 | < 2s（首屏） |

### 5.2 可靠性

1. Agent 进程异常退出时，平台自动检测并标记为 FAILED
2. **启动僵尸清理**：平台重启时自动清理异常状态的 AgentInstance 和分身
3. 沙箱磁盘空间不足时，提前告警并优雅终止 Agent
4. LLM API 调用失败时，自动重试（指数退避），支持 Fallback 到备用模型
5. 平台崩溃重启后，可恢复运行中的工作流执行状态
6. **错误分类**：Agent 失败时自动分类错误（dispatch_config / dispatch_runtime / agent / sandbox / network / other），便于前端快速过滤

### 5.3 安全

1. API Key 加密存储，明文不可泄露
2. Agent 沙箱严格隔离，禁止访问 `/etc/passwd`、其他租户目录等敏感路径
3. Agent 网络访问可控（可配置白名单/黑名单）
4. 所有操作记录审计日志，保留 180 天
5. GitHub OAuth 授权范围最小化（只申请必要的 repo 权限）
6. **项目级并发限制**：每个项目可配置最大并发 Agent 数，防止资源耗尽
7. **Prompt 注入防护**：模板变量自动转义，用户输入禁止直接拼接到系统 Prompt

### 5.4 可扩展性

1. LLM Provider 插件化：新增 Provider 无需修改核心代码
2. Agent CLI 适配器插件化：支持未来接入新的 CLI 工具
3. 产物处理器插件化：支持自定义产物解析和展示
