# STANDARD-08: 全栈开发规范

> **版本**: v1.0
> **生效日期**: 2026-04-03
> **强制级别**: 🔴 所有涉及前后端交互的功能必须遵守

---

## 1. 目的

解决前后端开发中常见的"半拉子工程"问题：

- 后端 API 已完成，前端未开始，功能无法使用
- 前端页面已开发，后端接口未实现，功能无法使用
- 前后端接口契约不一致，联调时发现大量问题
- 功能开发周期长，前后端各自为战，无法验收

---

## 2. 核心原则

### 2.1 功能完整性原则

**任何涉及前后端交互的功能，必须保证前后端均完成开发才算完成。**

```
❌ 不允许的情况：
- 后端 API 已合并，前端未开始
- 前端页面已开发，后端接口未完成
- 接口契约不一致，联调失败

✅ 允许的情况：
- 前后端均已完成，联调通过
- 纯后端功能（如批处理、定时任务）
- 纯前端功能（如静态页面、UI组件）
```

### 2.2 同步开发原则

| 阶段 | 后端 | 前端 | 说明 |
|------|------|------|------|
| 设计阶段 | API 设计 | 接口确认 | 双方确认 API 契约 |
| 开发阶段 | 并行开发 | 并行开发 | Mock 数据辅助开发 |
| 联调阶段 | 接口调试 | 页面调试 | 必须验证可用 |
| 验收阶段 | 一起验收 | 一起验收 | 功能完整可用 |

---

## 3. 开发流程

### 3.1 功能拆分要求

涉及前后端交互的功能（PRD），必须拆分为以下子任务：

```
PRD-XXX: 用户认证功能
├── BE-XXX-1: 后端 - 登录 API 开发
├── BE-XXX-2: 后端 - Token 验证中间件
├── FE-XXX-1: 前端 - 登录页面开发
├── FE-XXX-2: 前端 - 登录状态管理
└── INTEGRATION-XXX: 前后端联调验证
```

**规则**：
- `INTEGRATION-XXX` 任务未完成，整个功能不算完成
- 禁止单独关闭 BE-XXX 或 FE-XXX 而不关闭 INTEGRATION-XXX

### 3.2 API 契约先行 + 契约锁定

在编码开始前，必须完成并评审 API 设计文档：

```markdown
## API 契约文档 (API-{功能}.md)

### 接口列表
| 接口 | 方法 | 路径 | 后端状态 | 前端状态 |
|------|------|------|---------|---------|
| 登录 | POST | /api/v1/auth/login | ⏳ 待开发 | ⏳ 待开发 |
| 登出 | POST | /api/v1/auth/logout | ⏳ 待开发 | ⏳ 待开发 |

### 请求/响应示例
...详细示例...

### 评审记录
- 后端评审人：___
- 前端评审人：___
- 评审日期：___
```

#### API 契约锁定机制（新增 — 2026-04-21）

契约评审通过后，双方签字确认，进入**锁定状态**：

```
状态：🔒 已锁定（Locked）
```

**锁定期间变更规则**：
1. **任何一方不得单方面修改契约**
2. 如需变更，必须在契约文档中用 `🔴 [变更]` 标记变更点
3. 变更方必须 @ 通知对方，对方确认后方可修改代码
4. 变更后必须重新走 mini 评审（5 分钟快速确认）
5. 联调时发现契约与代码不一致，**责任方是契约的修改方**

**变更记录格式**：
```markdown
### 变更记录
| 日期 | 变更人 | 变更内容 | 确认人 | 状态 |
|------|--------|---------|--------|------|
| 2026-04-21 | @后端A | 将 `status` 字段从 int 改为 string | @前端B | ✅ 已确认 |
```

### 3.3 Mock 开发策略

为了并行开发，允许使用 Mock 数据：

```javascript
// 前端开发时使用 Mock
const useMock = process.env.NODE_ENV === 'development';

const login = async (credentials) => {
  if (useMock) {
    return mockLogin(credentials); // Mock 响应
  }
  return api.post('/api/v1/auth/login', credentials);
};
```

**Mock 数据必须基于已评审的 API 契约文档。**

## 4. 验收标准

### 4.3 联调证据要求（新增 — 2026-04-21）

联调通过**必须提供以下证据之一**，口述"我测过了"不算：

| 证据类型 | 要求 | 优先级 |
|---------|------|--------|
| **联调录屏/GIF** | 展示完整链路：前端页面操作 → Network 面板显示 200 → 后端日志显示处理 → 前端渲染正确结果 | 首选 |
| **联调截图** | 至少 2 张：① 前端 Network 面板显示请求/响应 ② 后端日志显示处理成功 | 次选 |
| **联调报告** | `docs/verification/INTEGRATION-{功能ID}.md` 文件，含测试步骤、实际请求/响应、结果、双方签字 | 备选 |

**❌ 不接受的证据**：
- "我本地测过了"（口述，无截图）
- "Swagger 能调通"（未涉及前端真实调用）
- "单元测试通过了"（Mock 数据不等于真实联调）
- "Postman 能调通"（未涉及前端真实调用）

**联调报告模板**：
```markdown
# INTEGRATION-{功能ID} 联调报告

## 基本信息
- 功能：xxx
- 联调日期：YYYY-MM-DD
- 后端开发：@xxx
- 前端开发：@xxx

## 测试接口
| 接口 | 方法 | 路径 | 状态 |
|------|------|------|------|
| 登录 | POST | /api/v1/auth/login | ✅ 通过 |

## 测试步骤
1. 打开登录页面
2. 输入用户名/密码
3. 点击登录按钮

## 实际请求/响应
```json
// Request
{ "username": "test", "password": "123456" }

// Response (200)
{ "token": "eyJhbG...", "expiresIn": 3600 }
```

## 结果
- [x] 正常流程通过
- [x] 错误处理验证（密码错误返回 401）
- [x] 边界条件验证（空密码前端拦截 + 后端校验）

## 签字确认
- 后端开发：___（必须手写签名或 Git 账号 @）
- 前端开发：___（必须手写签名或 Git 账号 @）
```

### 4.1 功能完成检查清单

```markdown
## 功能验收清单

### 后端检查项
- [ ] API 已实现并部署到测试环境
- [ ] API 文档已更新（Swagger/OpenAPI）
- [ ] 单元测试覆盖率 ≥ 80%
- [ ] 集成测试通过

### 前端检查项
- [ ] 页面已实现并与设计稿一致
- [ ] 与后端 API 联调通过
- [ ] 单元测试通过
- [ ] E2E 测试通过（关键路径）

### 联调检查项
- [ ] 真实接口调用成功（非 Mock）
- [ ] 错误处理场景验证
- [ ] 边界条件验证
- [ ] 性能指标达标
- [ ] **联调证据已提交（录屏/截图/联调报告，见 §4.3）**

### 签字确认
- 后端开发：___（必须为真实人员，AI 不能代签）
- 前端开发：___（必须为真实人员，AI 不能代签）
- 产品经理：___
```

### 4.2 禁止提交的场景

以下情况禁止合并到主分支：

| 场景 | 示例 | 处理方式 |
|------|------|---------|
| 后端先行 | 后端 API 已合并，前端未开始 | PR 拒绝，等待前端 |
| 前端先行 | 前端页面已合并，后端未实现 | PR 拒绝，等待后端 |
| 接口不一致 | 后端返回字段与文档不一致 | PR 拒绝，修复后重新联调 |
| 未经验证 | 声称完成但未联调验证 | PR 拒绝，补充验证报告 |

## 5. 项目管理

### 5.1 任务状态追踪

在 PROJECT_STATUS.md 中必须体现前后端状态：

```markdown
| 功能 | ID | 后端状态 | 前端状态 | 联调状态 | 整体状态 |
|------|-----|---------|---------|---------|---------|
| 用户认证 | F-01 | ✅ 完成 | ✅ 完成 | ✅ 通过 | ✅ 完成 |
| 工作流管理 | F-02 | ✅ 完成 | ⏳ 开发中 | ⏳ 待开始 | 🔄 进行中 |
| 任务监控 | F-03 | ⏳ 待开始 | ⏳ 待开始 | ⏳ 待开始 | ⏳ 待开始 |
```

### 5.2 分支管理（强制）

涉及前后端的功能，**必须使用统一的功能分支**（强制，非建议）：

```bash
# ❌ 禁止：前后端各自分支
feature/PRD-001-backend
feature/PRD-001-frontend

# ✅ 强制：统一功能分支
feature/PRD-001-workflow-engine
```

**违规后果**：
- CI 检测到 PR 只包含后端或只包含前端代码，且该功能涉及前后端交互 → **自动拒绝合并**
- 功能未完成（缺少前端或后端）就尝试合并 → **PR 被标记为 blocked**

**特殊情况**：纯后端或纯前端的功能，必须在 PRD 中明确标注，经 Tech Lead 确认后可以使用单独分支。

## 6. 联调规范

### 6.1 联调启动条件

前后端联调必须满足以下条件才能开始：

- [ ] 后端 API 已部署到测试环境
- [ ] 前端页面已完成开发
- [ ] API 契约文档双方已确认
- [ ] 双方已约定联调时间

### 6.2 联调验证流程

```bash
# 1. 后端先自测
./mvnw test
./scripts/api-self-test.sh  # 使用 httpie 或 curl 验证 API

# 2. 前端切换为真实接口
# .env.local
VITE_API_BASE_URL=http://localhost:8080/api/v1
VITE_USE_MOCK=false

# 3. 双方一起验证
# - 正常流程
# - 错误处理
# - 边界条件

# 4. 记录验证结果
# docs/verification/API-001-verification.md
```

### 6.3 联调问题处理 + 升级机制

| 问题类型 | 处理方式 | 责任人 |
|---------|---------|--------|
| 接口不符合契约 | 后端修复 | 后端开发 |
| 前端调用方式错误 | 前端修复 | 前端开发 |
| 契约文档遗漏 | 双方协商更新文档 | 双方 |
| 需求理解不一致 | 找产品经理确认 | 产品经理 |

#### 联调问题升级机制（新增 — 2026-04-21）

如果联调问题在 **2 小时内** 无法双方协商解决，必须升级：

```
0-2 小时：双方自行协商
    ↓ 未解决
2-4 小时：@Tech Lead 介入协调
    ↓ 未解决
4 小时后：@PM + @Tech Lead 一起决策
```

**升级触发条件**：
- 接口契约理解不一致（后端说 A，前端说 B）
- 技术方案冲突（如数据格式、状态码定义）
- 一方修改了契约但未通知对方
- 任何一方拒绝配合联调

**禁止行为**：
- ❌ 联调卡住了也不上报，各自干等
- ❌ 绕过对方直接修改契约或接口
- ❌ 以"先合并再修"为理由跳过联调

## 7. 工具支持

### 7.1 推荐工具

| 工具 | 用途 | 阶段 |
|------|------|------|
| Swagger/OpenAPI | API 文档 | 设计、开发 |
| Postman/Apifox | API 测试 | 开发、联调 |
| MSW (Mock Service Worker) | 前端 Mock | 开发 |
| Storybook | 前端组件独立开发 | 开发 |

### 7.2 API 文档生成

后端必须提供自动生成的 API 文档：

```java
// SpringDoc OpenAPI 配置
@Configuration
public class OpenApiConfig {
    
    @Bean
    public OpenAPI customOpenAPI() {
        return new OpenAPI()
            .info(new Info()
                .title("AI Dev Platform API")
                .version("1.0.0"));
    }
}
```

访问 `http://localhost:8080/swagger-ui.html` 查看文档。

## 8. 违规处理

| 违规级别 | 行为 | 处理方式 |
|---------|------|---------|
| 🔴 严重 | 后端 API 未联调就合并到主分支 | PR 拒绝，撤销合并 |
| 🔴 严重 | 前端页面未联调就合并到主分支 | PR 拒绝，撤销合并 |
| 🔴 严重 | **声称"完成"但未提供联调证据** | **标记为未完成，强制补充联调证据后才能合并** |
| 🔴 严重 | **前后端不在同一功能分支开发** | **PR 拒绝，要求重新提交到统一分支** |
| 🔴 严重 | **单方面修改 API 契约未通知对方** | **回滚修改，重新走契约变更流程** |
| 🟠 重要 | 接口契约未评审就开始编码 | 暂停开发，补评审 |
| 🟠 重要 | 联调发现问题不记录不修复 | 暂停合并，修复完成 |
| 🟠 重要 | **联调问题 2 小时内不上报升级** | **警告，计入个人/AI 交付质量记录** |
| 🟡 一般 | 功能状态未及时更新 | 提醒更新 PROJECT_STATUS.md |

## 9. 特殊情况

### 9.1 纯后端功能

以下功能不受此规范约束：
- 定时任务
- 批处理作业
- 内部数据迁移工具
- 命令行工具

**但必须在 PRD 中明确标注"纯后端功能"**。

### 9.2 纯前端功能

以下功能不受此规范约束：
- 静态展示页面
- UI 组件开发
- 纯前端交互（如拖拽排序，数据来自前端状态）

**但必须在 PRD 中明确标注"纯前端功能"**。

## 10. 检查清单

创建涉及前后端的功能时，使用以下检查清单：

```markdown
## PRD-XXX 功能创建检查清单

### 需求阶段
- [ ] 明确是否为纯后端/纯前端/前后端交互
- [ ] 创建 API 契约文档
- [ ] 前后端负责人确认 API 设计
- [ ] **双方签字确认 API 契约（进入锁定状态）**

### 开发阶段
- [ ] 拆分为 BE-XXX、FE-XXX、INTEGRATION-XXX 任务
- [ ] 更新 PROJECT_STATUS.md
- [ ] **使用统一功能分支（强制）**
- [ ] **如契约需变更，标记变更并通知对方确认**

### 联调阶段
- [ ] 后端 API 已部署到测试环境
- [ ] 前端页面已完成开发
- [ ] **双方按契约进行真实接口调用验证**
- [ ] **提供联调证据（录屏/截图/联调报告）**
- [ ] **双方签字确认联调通过（AI 不能代签）**

### 验收阶段
- [ ] 后端自测通过
- [ ] 前端自测通过
- [ ] **联调验证通过（必须有证据，不能口述）**
- [ ] 更新功能状态为"完成"
```
