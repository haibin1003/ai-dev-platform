# STANDARD-05: 工程状态维护规范

> **版本**: v1.0
> **生效日期**: 2026-04-03
> **强制级别**: 🔴 必须遵守

---

## 1. 目的

工程状态文档是让后续开发者快速入场、避免重复劳动的关键。本规范确保：

- 任何时间点，项目状态清晰可查
- 新开发者能在 5 分钟内了解项目进展和当前焦点
- 已完成/待开发功能有明确记录
- 开发者能快速找到继续工作的起点

---

## 2. 必须维护的文档

| 文档 | 位置 | 格式 | 更新时机 |
|------|------|------|---------|
| 工程状态报告 | `PROJECT_STATUS.md` | Markdown | 每次状态变更时 |
| 功能清单 | `PROJECT_STATUS.md` | Markdown 表 | PRD 审批通过时 |

---

## 3. 状态定义

### 3.1 里程碑状态

| 状态 | 含义 |
|------|------|
| ⏳ 待开始 | 尚未开始 |
| 🔄 进行中 | 正在开发 |
| ✅ 完成 | 已完成并通过评审 |
| ❌ 废弃 | 因故废弃 |

### 3.2 功能状态

| 状态 | 含义 |
|------|------|
| ⏳ 待开发 | 排队中，未开始 |
| 🔄 开发中 | 正在实现 |
| ✅ 已完成 | 开发+测试+评审通过 |
| ❌ 已废弃 | 因故废弃 |

### 3.3 阻塞项标记

在 `PROJECT_STATUS.md` 中使用 `- [ ]` 标记未解决的阻塞项，使用 `- [x]` 标记已解决的阻塞项。

---

## 4. 更新触发条件

### 4.1 必须立即更新的场景

| 事件 | 更新内容 | 更新人 |
|------|---------|--------|
| 新 PRD 审批通过 | 在待开发功能表添加条目 | 需求分析师/AI |
| ARCH 评审通过 | 更新里程碑状态 | 架构师/AI |
| DESIGN 评审通过 | 更新里程碑状态 | 开发负责人/AI |
| feature 分支合并 | 更新已完成功能表 | PR 合并者 |
| 发现新的阻塞项 | 在当前焦点添加阻塞项 | 发现者 |
| 阻塞项解决 | 在 PROJECT_STATUS.md 标记为已完成 | 解决者 |

### 4.2 建议更新的场景

| 场景 | 更新内容 |
|------|---------|
| 开发环境检查失败 | 更新开发环境状态表 |
| 测试覆盖率下降 | 在备注中说明原因 |
| 分支长时间无进展 | 标注可能的风险 |

## 5. AI 开发强制要求

### 5.1 入场强制检查

**每次开始工作前，AI 必须：**

1. **阅读 `PROJECT_STATUS.md`**
   - 了解当前里程碑状态
   - 确认所开发功能的位置（待开发/进行中/已完成）
   - 识别当前阻塞项

2. **验证开发环境状态**
   ```bash
   ./mvnw clean compile
   ./mvnw checkstyle:check
   ./mvnw spotbugs:check
   ```

3. **检查是否有新变更**
   ```bash
   git pull origin master
   ```

### 5.2 开发过程要求

**AI 在开发过程中必须：**

- [ ] 遵守 `AGENTS.md` 规定的流程（PRD → ARCH → DESIGN → DEV → CR → QA）
- [ ] 在 `PROJECT_STATUS.md` 中更新所负责功能的当前状态
- [ ] 发现阻塞项时，立即在 `PROJECT_STATUS.md` 中记录
- [ ] 功能完成后，更新功能清单中的状态和测试覆盖率

### 5.3 提交前检查

**PR 提交前，AI 必须验证：**

- [ ] `PROJECT_STATUS.md` 已更新（功能状态、里程碑状态）
- [ ] 无未记录的阻塞项
- [ ] 所有检查通过（test、checkstyle、spotbugs、jacoco）

## 6. 文档格式规范

### 6.1 PROJECT_STATUS.md 必需章节

```markdown
# 工程状态报告

## 📊 当前里程碑      ← 必须有
## 🎯 当前开发焦点    ← 必须有
## ✅ 已完成功能       ← 必须有
## 📋 待开发功能       ← 必须有
## 🔧 开发环境状态     ← 必须有
## 🗂️ 文档矩阵        ← 必须有
## 🚀 快速入场指南     ← 必须有
## 📝 变更记录         ← 必须有
```

### 6.2 功能清单格式

```markdown
| 功能 | ID | PRD | 优先级 | 估计工时 | 依赖 | 备注 |
|------|-----|-----|--------|---------|------|------|
| 工作流定义管理 | F-01 | PRD-001 | P0 | - | ARCH 设计 | YAML/JSON 解析 |
```

### 6.3 里程碑格式

```markdown
| 阶段 | 状态 | 完成日期 | 说明 |
|------|------|---------|------|
| ARCH-001 工作流引擎架构 | ✅ 完成 | 2026-04-03 | AI 评审通过 |
```

## 7. 违规处理

| 违规级别 | 行为 | 处理方式 |
|---------|------|---------|
| 🔴 严重 | 入场前未阅读 PROJECT_STATUS.md | 代码拒绝，必须先阅读并确认理解 |
| 🔴 严重 | 开发过程未更新工程状态 | PR 拒绝，必须补充更新 |
| 🟠 重要 | 阻塞项未记录 | 补充记录后才能继续 |
| 🟡 一般 | 变更记录不完整 | 建议补充，可接受但不推荐 |
